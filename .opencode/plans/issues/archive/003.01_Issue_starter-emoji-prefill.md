# Issue 003.01 — Starter emoji icons + prefill-not-send (Bundle A)

- **Status**: DONE (2026-07-08)
- **PRD**: 003_PRD_deeper-assistant-integration.md (Bundle A)
- **Dependencies**: none
- **Blocks**: —
- **Bundle**: A (Assistant QoL)

## Goal

Starter buttons render reliably (emoji-in-label) and, on click: inject context (for task starters) AND prefill the composer with an editable seed the user sends manually — instead of today's fire-and-send `#12 — let's work on this.`

## Why

Today starters auto-send a canned phrase, leaving no room for the user to add context before sending. Icons (Lucide names) also don't render reliably. The user wants to click a starter, see the task context land, then type their angle and send.

## Scope (TDD-first)

### Spike gate (do this FIRST)

`cl.Starter` has no native prefill API — it sends on click. Confirm whether Chainlit's composer is reachable from a custom JS event dispatched via `chat/public/simpler-bridge.js` (the same bridge that already forwards the space filter). If reachable, proceed with the prefill approach below. If not, fall back to: starter click copies the seed to the clipboard + shows a toast "pasted into composer — edit and send" (lower-fi but unblocks the feature).

Record the spike outcome in the issue's final comment.

### Failing tests first (`tests/test_chat_*.py`)

1. `test_build_starters_emoji_in_label` — every starter spec's `label` starts with an emoji char; no `icon` key in the spec dict (or it's empty). Drives `chat/commands.py::build_starters` changes.
2. `test_task_starter_has_command_and_seed` — a task starter spec carries `'command': 'task'` AND a new `'prefill': '#<id> — '` field (the editable seed, not sent). Generic starters carry `'prefill': <their current message>` and `'command': None`.
3. `test_starter_seed_not_in_message` — the `message` field is removed from starter specs (clicking no longer auto-sends a model message); the seed lives only in `prefill`.

### Implementation

- `chat/commands.py::build_starters`:
  - Replace the `'message'` field with `'prefill'` (the editable text).
  - Task starters: `'prefill': f"#{task['id']} — "`, keep `'command': 'task'` (context injection still fires).
  - Generic starters: `'prefill': <their current prompt text>, 'command': None`.
  - Prefix every `label` with an emoji: `▶` for tasks, `✨` for generics. Drop the `icon` field.
- `chat/chainlit_app.py::starters`:
  - Build `cl.Starter(label=spec['label'], message=spec['prefill'], command=spec.get('command'))` — but `message` now means "prefill" only because we intercept the send. See the bridge change.
  - Wire a custom event so `simpler-bridge.js` can read the prefill and set the composer text after the `/task` injection.
- `chat/public/simpler-bridge.js`:
  - Listen for the starter-click custom event; set the Chainlit composer input to the prefill text WITHOUT submitting.
  - The `/task` context injection still fires server-side (unchanged from today).

## Closing note — spike outcome + implementation (2026-07-08)

**Spike outcome: prefill approach CONFIRMED feasible** (verified against the
shipped Chainlit 2.11.1 frontend source, extracted from the bundle source
maps — no fallback needed):

- The composer is a controlled React textarea with a stable `id="chat-input"`;
  external JS can set it with the native value setter + a bubbling `input`
  event (React onChange fires, state updates).
- Starter buttons carry stable ids (`starter-<label-slug>`); a capture-phase
  document click listener runs before React's root-delegated handler, so
  `stopImmediatePropagation()` fully suppresses the auto-send.
- Two-way JS⇄backend messaging already exists: window `message` events are
  relayed to `@cl.on_window_message` (AppWrapper.tsx — the space filter uses
  this), and `cl.send_window_message()` posts back to `window.parent`
  (reachable same-origin from the bridge).

**Implemented flow**: bridge intercepts the starter click (nothing is sent),
posts `{type:'simpler-starter-click', label}` to the backend;
`on_starter_click` matches the spec by label (`commands.starter_by_label`),
runs the `/task` injection for task starters (system message lands in the
thread), and answers `{type:'simpler-starter-prefill', prefill}`; the bridge
writes the seed into the composer (cursor at end, not sent). If the bridge is
absent the `cl.Starter.message` fallback degrades to the old fire-and-send
behavior with the bare seed.

Tests: 4 new in `tests/test_chat_commands.py` (emoji labels, prefill+command
fields, no `message` key, label lookup); existing starter tests updated.
