# Issue 003.06 — Skills authoring: agent tool + in-app form (Bundle B)

- **Status**: DONE (2026-07-08)
- **PRD**: 003_PRD_deeper-assistant-integration.md (Bundle B)
- **Dependencies**: 003.02 (instance skills dir)
- **Blocks**: 003.07 (settings panel lists/edits skills)
- **Bundle**: B

## Goal

The model can create skills via a `create_skill` tool (writes `instance/assistant/skills/<name>/SKILL.md`). The user can create/edit/delete skills from an in-app form. Skills dir resolution: instance dir (read+write) first, bundled `chat/skills/` (read-only fallback) second — both listed in the panel.

## Why

Today skills are on-disk packages under `chat/skills/` only — read-only, no in-app authoring. The user wants the agent to draft skills on request AND to edit/add them manually, with a path to manage skill files directly via the file browser later (Bundle C dovetail).

## Scope (TDD-first)

### Failing tests first (`tests/test_chat_skills.py` — extend existing, or new file)

1. `test_create_skill_writes_instance_dir` — `skills.create_skill(name='myflow', description='…', body='# Steps\n…')` creates `instance/assistant/skills/myflow/SKILL.md` with the front matter + body; returns a success message.
2. `test_create_skill_rejects_collision` — creating a skill whose name exists in the instance dir returns a `TOOL ERROR:` message (no overwrite).
3. `test_create_skill_rejects_bundled_name_collision` — creating a skill whose name matches a bundled one (e.g. `braindump`) returns a TOOL ERROR suggesting a different name (bundled skills are read-only).
4. `test_list_skills_merges_instance_and_bundled` — `list_skills()` returns instance skills (marked `source: 'instance'`) AND bundled ones (marked `source: 'bundled'`), instance-first, no duplicate names (instance wins on name clash).
5. `test_delete_skill_instance_only` — `delete_skill('myflow')` removes the instance dir; `delete_skill('braindump')` returns a TOOL ERROR (bundled = undeletable).
6. `test_update_skill_body` — `update_skill(name, body)` rewrites the instance SKILL.md body (front matter preserved); rejects bundled.
7. `test_use_skill_tool_registered` — the `use_skill` tool is still registered (no regression).

### Implementation

- `chat/skills.py`:
  - `skills_dir()` → `instance_skills_dir()` (from `assistant_settings`); `bundled_skills_dir()` separate.
  - `list_skills()` merges both, instance-first, with a `source` field.
  - New `create_skill(name, description, body, files=None)` — writes `instance/assistant/skills/<name>/SKILL.md` (front matter + body), validates name (no path traversal, no collision with instance OR bundled).
  - New `update_skill(name, body, description=None)` — rewrites the body in an instance skill (creates the instance copy if the user is "editing" a bundled one → copy-to-instance-then-edit semantics; OR rejects with a hint; **decided: copy-to-instance-then-edit** — matches "down the line I'll manage files directly" intent).
  - New `delete_skill(name)` — instance only.
- `chat/toolbox.py` / `chat/skills.py::register`:
  - Register `create_skill`, `update_skill`, `delete_skill` as native tools alongside `use_skill`.
- `chat/commands.py::_handle_skill` (the `/skill` command): list merged sources so the model sees both.
- No in-app FORM UI in this issue — that's the settings panel (003.07). Here we build the backend + tools only.

### Copy-to-instance-then-edit semantics

When the user (or model via `update_skill`) edits a bundled skill by name, the system: (1) copies `chat/skills/<name>/` → `instance/assistant/skills/<name>/`, (2) applies the edit to the instance copy, (3) subsequent `list_skills` shows the instance copy (which shadows the bundled one by name). This makes bundled skills effectively "forkable" without ever mutating the shipped read-only set.
