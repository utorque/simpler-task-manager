# Issue 003.04 — Migrate model picker from Chat Profiles to Modes (Bundle A/B)

- **Status**: DONE (2026-07-08)
- **PRD**: 003_PRD_deeper-assistant-integration.md (Bundle A + B)
- **Dependencies**: 003.02 (reads `models.json`)
- **Blocks**: 003.07 (settings panel manages the model list)
- **Bundle**: A (core), B (settings integration)

## Goal

Replace `@cl.set_chat_profiles` with `cl.context.emitter.set_modes` for the model picker. Model selection becomes a per-chat Mode (id `model`) that persists across messages and is switchable mid-conversation. Model list source: `instance/assistant/models.json` (written by the settings panel in 003.07), falling back to `CHAT_MODELS` env → `AI_MODEL`.

## Why

Chat Profiles force one model-per-chat-history bucket and URL-only icons (no emoji/lucide). Modes support multiple picker categories, lucide icons, mid-chat switching, and are extensible to reasoning effort (003.05) and future personas. This is the more flexible primitive.

## Scope (TDD-first)

### Failing tests first (`tests/test_chat_model_modes.py` — new file)

1. `test_model_list_reads_instance_json` — given `instance/assistant/models.json` contains `["gpt-4o", "claude-3-7-sonnet"]`, `available_models()` returns exactly that list; the first is marked default.
2. `test_model_list_falls_back_to_env` — given no `models.json` (or empty), `available_models()` falls back to `CHAT_MODELS` env split, then `AI_MODEL` env, then `["gpt-3.5-turbo"]` (current behavior).
3. `test_build_model_mode_options` — `build_modes(model_kind='model')` returns a list of `ModeOption`-shaped dicts, one per model, first with `default=True`, each with a lucide `icon` name.
4. `test_selected_model_read_from_message_modes` — given an incoming `cl.Message` with `msg.modes = {'model': 'gpt-4o'}`, `current_model(msg)` returns `'gpt-4o'`; given no modes, falls back to the default (first in list).
5. `test_on_chat_start_sets_modes` — `on_chat_start` calls `cl.context.emitter.set_modes` with at least the `model` category (mock the emitter).

### Implementation

- `chat/assistant_settings.py` (from 003.02): add `available_models()` — reads `models.json` (instance), falls back to `CHAT_MODELS`/`AI_MODEL`/default. (The `read_models()` from 003.02 may already cover this; add the fallback chain here.)
- New module `chat/modes.py` (UI-agnostic, testable without Chainlit):
  - `MODEL_MODE_ID = 'model'`
  - `build_model_mode_options(models: list[str]) -> list[dict]` — `[{id, name, description, icon, default}]`, first `default=True`, `icon='cpu'` (lucide).
  - `current_model_from_modes(modes: dict | None, default: str) -> str` — reads `modes.get('model')`, falls back to `default`.
- `chat/chainlit_app.py`:
  - REMOVE `@cl.set_chat_profiles` and the `current_model()` that reads `cl.user_session.get('chat_profile')`.
  - In `on_chat_start`: `await cl.context.emitter.set_modes(modes.build_model_mode_options(assistant_settings.available_models()))`.
  - In `on_message`: `model = modes.current_model_from_modes(getattr(message, 'modes', None), default=assistant_settings.available_models()[0])`, pass to `run_agent(model=model, ...)`.
  - Keep `get_llm()` unchanged.

## Acceptance criteria

## Closing note (2026-07-08)

Chat Profiles are gone (pinned by a source-inspection test): the model picker
is now a `model` Mode built in `chat/modes.py` (UI-agnostic dicts →
`cl.Mode`/`cl.ModeOption` in `chainlit_app.build_chat_modes()`), published by
`publish_modes()` on BOTH `on_chat_start` and `on_chat_resume` (resumed
threads get pickers too — their old profile selection falls back to default,
no backfill, as per PRD risk note). Selection is read per message via
`modes.current_model_from_modes(message.modes, default)`. Model list:
`assistant_settings.available_models()` = instance `models.json` when
non-empty, else the CHAT_MODELS/AI_MODEL env chain. Modes API verified
against installed Chainlit 2.11.1 (`chainlit/mode.py`,
`emitter.set_modes`, `Message.modes`). Tests: 6 in
`tests/test_chat_model_modes.py`.
