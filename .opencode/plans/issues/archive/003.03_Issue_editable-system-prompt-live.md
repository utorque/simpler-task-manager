# Issue 003.03 — Editable system prompt (live, instance-override) (Bundle B)

- **Status**: DONE (2026-07-08)
- **PRD**: 003_PRD_deeper-assistant-integration.md (Bundle B)
- **Dependencies**: 003.02 (instance dir foundation)
- **Blocks**: 003.07 (settings panel exposes the editor)
- **Bundle**: B (Prompt/skill authoring & prompt layering)

## Goal

The base system prompt is read **per message** (not module-global) from `instance/assistant/system.md` when present, else the shipped `chat/prompts/system.md`. Edits via the app take effect on the next user message with no restart.

## Why

Today `SYSTEM_PROMPT = load_system_prompt()` at module import (line 53 of `chainlit_app.py`) means: (a) edits to the source file need a process restart, (b) `git pull` can silently clobber the user's customizations. The per-turn layering (date/spaces/tools/skills) is already last-instant, but the base prompt is stale-across-edits.

## Scope (TDD-first)

### Failing tests first (`tests/test_chat_system_prompt.py` — new file)

1. `test_build_system_prompt_reads_instance_override` — given `instance/assistant/system.md` contains `# My custom prompt`, `build_system_prompt(toolbox=None)` returns text starting with `# My custom prompt`.
2. `test_build_system_prompt_falls_back_to_shipped` — given no instance override, `build_system_prompt(toolbox=None)` returns the shipped `chat/prompts/system.md` content as the leading block.
3. `test_build_system_prompt_includes_dynamic_layers` — the assembled prompt contains the current date line, the spaces guidance block (when `simpler_client.configured()` + spaces returned), the tools list (when a toolbox with specs is passed), and the skills section (when skills exist). Verifies the per-turn layering stays intact after the refactor.
4. `test_load_system_prompt_pure_function` — `load_system_prompt()` is importable without Chainlit's runtime and returns a non-empty string.

### Implementation

- `chat/chainlit_app.py`:
  - Remove the module-global `SYSTEM_PROMPT = load_system_prompt()` (line 53).
  - In `build_system_prompt(toolbox)`, call `load_system_prompt()` (now from `chat.assistant_settings`/a small helper) per invocation, so edits are live.
  - Keep `load_system_prompt()` as a thin wrapper over `assistant_settings.system_prompt_path()` that reads the file (try/except `FileNotFoundError` → shipped default).
- Do NOT change the rest of `build_system_prompt`'s assembly (date, spaces, tools, skills) — only the base-prompt source becomes per-call.

## Acceptance criteria

- [ ] All four failing tests pass.
- [ ] Editing `instance/assistant/system.md` between two user messages changes the base prompt the model sees on the second message (no restart).
- [ ] Deleting `instance/assistant/system.md` reverts to the shipped default on the next message.
- [ ] The per-turn layering (date/spaces/tools/skills) is unchanged — verified by `test_build_system_prompt_includes_dynamic_layers`.

## Out of scope

- The in-app editor UI (issue 003.07).
- "Reset to shipped" button (issue 003.07, calls `assistant_settings.reset_system_prompt()`).
- Documenting the global-vs-injection boundary in `system.md` itself (the file content is the user's to edit; the boundary is enforced by `build_system_prompt`'s structure, which is unchanged here).


## Closing note (2026-07-08)

`load_system_prompt()` moved to `chat/assistant_settings.py` (Chainlit-free,
resolves instance override → shipped default per call); the module-global
`SYSTEM_PROMPT` in `chainlit_app.py` is gone and `build_system_prompt()`
re-reads the base prompt every invocation. One extra hardening beyond the
spec: `selected_space_ids()` now tolerates a missing Chainlit session context
(returns None) so `build_system_prompt()` is callable from Flask routes —
needed by 003.07's composition viewer. Tests: 5 in
`tests/test_chat_system_prompt.py` (override, fallback, live edit between
calls, dynamic layers intact, pure-function import).
