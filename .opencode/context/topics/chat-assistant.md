# Chat Assistant (embedded first-party Chainlit)

> The `chat/` package — a Chainlit 2 app mounted same-origin at `/assistant` by `asgi.py`, replacing the former Hermes reverse-proxy approach (2026-07). It shares Simpler's `.env` (AI provider, secret key, API token) so there is one login and one provider config for the whole product. Everything under `chat/` is importable without Chainlit's runtime, so the pure logic stays unit-testable with plain pytest. Deep module: the agent loop + toolbox + providers are the reusable core; `chainlit_app.py` is wiring only.

## Mounting & identity

- `asgi.py` is the canonical entrypoint: a **FastAPI umbrella** that `mount_chainlit(app, target=chat/chainlit_app.py, path='/assistant')` (native ASGI, websockets stream) and mounts the Flask app at `/` via a2wsgi. `python src/app.py` runs Flask alone (no Assistant tab) for quick edits.
- **No second login**: the assistant is same-origin with Simpler, so the Flask session cookie rides along on every request. `chat/auth_bridge.py::is_authenticated()` validates a signed, logged-in session directly from the cookie signature. Single-user app ⇒ one fixed Chainlit identifier `"owner"`, so all chat history belongs to the one real user. There is no Chainlit user-management UI; `chainlit_app.py`'s `@cl.password_auth_callback` accepts any password as long as `is_authenticated()` is satisfied (the Flask login IS the gate).
- **Branding**: Chainlit's `GET /logo` serves `{CHAINLIT_APP_ROOT}/public/logo_{light,dark}.*` when present, so `chat/public/logo_light.svg` / `logo_dark.svg` (bolt + "Simpler" wordmark, mirroring the shell navbar's `fa-bolt` brand) replace the Chainlit logo everywhere it renders — header and the 200px welcome-screen mark. `simpler.css` hides the owner avatar menu (`#user-nav-button`) and the "Built with Chainlit" link: the shell already owns identity and logout.
- `chat/settings.py::ensure_chainlit_env()` sets `CHAINLIT_APP_ROOT=chat/` (so Chainlit's config/translations/files scaffold under `chat/`, not the cwd) and derives `CHAINLIT_AUTH_SECRET` from `SECRET_KEY` (overridable). It MUST run before anything imports `chainlit` — `asgi.py` and `chat/__init__.py` both call it at import time.

## Model ⇄ tools loop

- `chat/providers.py` — `LLMClient` speaks both **OpenAI** and **Anthropic** wire formats, chosen by a `provider_kind(base_url)` URL heuristic (`'anthropic'` if the base URL contains `anthropic`, else `'openai'`). `stream_chat(model, system, messages, tools, on_text)` streams text deltas through the callback and returns a `ChatResult` (full text + parsed `ToolCall`s) once the model finishes. Tool-call argument parsing is fault-tolerant (`_parse_arguments` handles malformed JSON). `provider_kind` is the only seam that decides wire format; everything upstream is provider-agnostic and works with any OpenAI-compatible endpoint (Infomaniak, Mistral, OpenAI) or Anthropic.
- `chat/agent.py::run_agent(llm, model, system, history, toolbox, hooks, max_rounds=8)` — the UI-free agent loop: stream a round → if no tool calls, append the assistant message and return; otherwise execute each tool call, append `role:'tool'` results, loop. `AgentHooks` (`on_text`, `on_round_end`, `on_tool_start`, `on_tool_end`) is the observation surface — the Chainlit layer renders from it, tests record from it. On budget exhaustion with tool calls still pending, one last text-only round emits a `MAX_ROUNDS_NOTE` summarizing progress. The transcript is the canonical OpenAI-style message list both wire formats consume.
- **Model picker (Modes)**: chat-bar Mode pickers, published by `chainlit_app.publish_modes()` on chat start AND resume — `chat/modes.py` builds the option dicts (UI-agnostic, Chainlit-free). Two categories: `model` (list from `instance/assistant/models.json` when non-empty — the settings panel writes it — else the `CHAT_MODELS`/`AI_MODEL` env chain, via `assistant_settings.available_models()`) and `reasoning` (levels from `instance/assistant/reasoning.json`, default `low/medium/high`, `medium` pre-selected). Both are read per message from `message.modes` and switchable mid-conversation. Reasoning is applied at the provider wire: `providers.apply_reasoning_effort()` maps to OpenAI `reasoning_effort` / Anthropic `thinking` budgets (2048/8192/16384; `max_tokens` bumped above the budget), no-op for `none`/unset. Chat Profiles are gone; pre-Modes threads fall back to the defaults.
- **Editable system prompt**: the base prompt is read PER MESSAGE by `assistant_settings.load_system_prompt()` — `instance/assistant/system.md` (in-app override, survives upgrades) when present, else the shipped `chat/prompts/system.md`. `build_system_prompt()` joins the layers produced by `build_system_prompt_layers()` (base / datetime / spaces guidance / tools / skills — each with metadata + exact text), which also feeds the settings panel's read-only composition viewer so the two can't drift.

## Toolbox

- `chat/toolbox.py::Toolbox` — the tool registry the agent calls into: `specs()` returns the OpenAI-style tool specs for the model, `execute(name, args)` runs a call. Registered via three paths: `add_native` (in-process Python fn), `add_mcp_server` (pre-integrated HTTP MCP server — specs listed once at startup), `add_mcp_session` (a UI-added MCP live session from `on_mcp_connect`). Names are namespaced as `<server>:<tool>` for MCP-sourced tools to avoid collisions.
- **Pre-integrated MCP servers** (configured in `chainlit_app.py::on_chat_start`):
  - **Simpler sidecar** (`SIMPLER_MCP_URL` → `mcp_server/`): ~26 typed tools wrapping `/api/*` with the `API_TOKEN` bearer header. Mutations are audited `actor='agent'`. `get_workspace_summary` is the conversation opener (sends the doing/todo/blocked board + notes + spaces as context). Tools mirror user intents: `create_task` / `update_task` / `move_task` / `delete_task` / `add_subtask` / `toggle_freeze` / `freeze_day` / `run_schedule` / `update_space_context` / `create_note` / `append_to_note` / `email_to_task_draft` / `draft_tasks_from_text`, plus read-only `list_tasks` / `get_task` / `list_spaces` / `get_calendar` / `list_notes` / `get_note` / `list_changelog` / `list_mailboxes` / `list_mail` / `read_mail`. Mail is read-only by design.
  - **Execution sandbox** (`SANDBOX_MCP_URL` → `sandbox/`): isolated file + code-run tools. Own container (resource-capped, non-root, no network egress by default), shares ONLY the `assistant-workspace` volume with the web service. `CHAT_FILES_DIR=/workspace` on the web side so uploads land there and produced files are picked back up.
- **Native tools**: `chat/web_tools.py` (web search via `ddgs`, opt-in via `CHAT_WEB_TOOLS=1`), `chat/files.py` (uploaded-file ingestion from `CHAT_FILES_DIR` + the `attach_file_to_answer` delivery tool, see "File delivery"), `chat/skills.py` (skill loader + `create_skill`/`update_skill`/`delete_skill` authoring tools, see below), `chat/commands.py` (slash commands, see below).

## Skills, slash commands, starters

- **Skills**: markdown-directory skill packages, two roots merged instance-first by `chat/skills.py::list_skills()` (each entry carries `source: 'instance'|'bundled'`): `instance/assistant/skills/` (user-authored, read+write, survives upgrades; shadows bundled names) and the shipped read-only `chat/skills/` (`braindump`, `weekly-review`). Authoring: the model has `create_skill` / `update_skill` / `delete_skill` native tools, and the settings panel has a form calling the same functions through `/api/assistant/skills*`. Editing a bundled skill forks it (copy-to-instance-then-edit — the shipped set is never mutated); bundled skills are undeletable. Surfaced via the `/skill` command and the skill-pick UI.
- **Slash commands** (`chat/commands.py::handle_command`): inject workspace entities into the conversation context as a markdown block so the model can act on them without round-trips. Commands: `/task <id|title>`, `/note <id|title>`, `/tasks` (board), `/notes` (list), `/skill <name>` (inject a skill body). `is_known()` gates dispatch. The block is prepended as a system/context message, never stored as a user message.
- **Starters** (`chat/commands.py::build_starters`): starter buttons built from the user's Doing tasks plus generic ones; labels open with an emoji char (`▶`/`✨` — no icon-font dependency). Clicking does NOT auto-send: `simpler-bridge.js` intercepts the click (capture phase, before React) and posts the label to the backend (`on_starter_click` via `@cl.on_window_message`); task starters run the `/task <id>` injection (system message lands in the thread), then the backend answers with the editable `prefill` seed (`#<id> — `) which the bridge writes into the composer (native value setter + `input` event on `#chat-input`) — the user adds their angle and sends. Without the bridge, clicks degrade to sending the bare seed.
- **Starter freshness**: Chainlit ships starters inside the config it fetches ONCE per page load, so a board change left them stale until F5. `app.js::publishAssistantStartersRev()` writes a signature of the Doing set to `localStorage.assistantStartersRev` on every `loadTasks()`; the bridge compares it each tick and reloads the iframe **only while the welcome screen is up** (`#starters` present, composer empty, no pin in flight) — the one state where a reload costs nothing. Mid-conversation the starters aren't on screen anyway and the next New Chat gets them fresh.
- **Pin a task to the assistant**: the robot button on a board card (`.board-card-assist`, left of the priority badge) calls `app.js::pinTaskToAssistant()` → `localStorage.assistantPinnedTask` + switch to the Assistant tab. localStorage rather than postMessage because the iframe is lazy-loaded and the first pin predates the bridge. The bridge consumes the key, posts `simpler-pin-task` (re-posting each tick until the backend's prefill arrives — a cold iframe's socket may not be up yet, capped at 5 tries), and `chainlit_app.py::on_pin_task` runs the same `inject_and_prefill('task', '#<id> — ')` path as a starter click — so it works mid-conversation too.
- **Workspace context** (`chat/workspace.py`): turns tasks/notes/spaces into markdown blocks. House rule: whenever a TASK is injected, its linked note is injected too (`format_task` takes an optional `note`). `format_task_board` groups by status (`doing/todo/blocked/done`); `format_spaces_guidance` frames per-space `context_markdown` as guide-not-source (same framing as `src/prompt_context.py`'s `space_guidance_block`).

## Workspace files (Bundle C, PRD 003)

- **`/api/workspace/*` blueprint** (`src/routes/workspace.py`, Flask side): tree/mkdir/upload/download/save/delete/move over an allowlisted multi-root map (`workspace` → `chat.settings.files_dir()`, `skills` → instance skills dir, reserved). Traversal gate = `chat/files.py::resolve_under()` (realpath + prefix, null-byte/backslash/absolute rejection, symlink-escape refusal) — shared with the attach tool. `GET /api/workspace/files/<root>/<path>` is the sanctioned download URL; text serves inline, the rest as attachment. Upload caps at `WORKSPACE_MAX_UPLOAD_MB` (default 100).
- **WorkspaceView drawer** (`src/static/js/workspace.js`): persistent right drawer inside the Assistant tab (`.assistant-body` wraps iframe + `<aside>`, ~`min(380px, 20vw)`), toggled by the Files button or `W`. Lazy tree, breadcrumb, drag-drop upload, create folder, inline EasyMDE `.md`/`.txt` editing (autosave on blur), download, delete, drag-move. Open/closed + last path persist in localStorage. The settings modal overlays it.
- **Drag a path into the chat**: drawer rows drag with two flavours — `application/x-workspace-path` (in-drawer move, unchanged) and `text/plain` carrying the workspace-relative path (`dir/` for folders). A drag started in the shell still dispatches `dragover`/`drop` into the same-origin assistant iframe, so `chat/public/simpler-bridge.js` intercepts it in the **window capture phase** (ahead of Chainlit's react-dropzone, which would otherwise treat it as an upload) and inserts the path backticked at the composer caret via the native-value-setter + `input` trick. `body.simpler-path-dragover` (in `simpler.css`) marks the hover. That relative path is exactly what `sandbox__*` and `attach_file_to_answer` take.
- **File delivery** (issue 003.10): NO auto-attach of files touched during a turn. The model delivers explicitly: `attach_file_to_answer(path)` (validated by `resolve_under`, queued per message, flushed after `run_agent` as `cl.File` chips) or inline relative markdown links `/api/workspace/files/workspace/<rel>` — both taught in `chat/prompts/system.md` ("Delivering files").

## Assistant settings panel (Bundle B, PRD 003)

- **`/api/assistant/*` blueprint** (`src/routes/assistant_settings.py`): GET aggregate settings; PUT models / reasoning-levels (validated non-empty string lists → `instance/assistant/*.json`); GET/PUT/DELETE system-prompt (override write / reset-to-shipped); skills CRUD wrapping `chat.skills`. The composition viewer returns `build_system_prompt_layers()` output (spaces pre-fetched from the DB — no HTTP loop-back into our own server; `build_toolbox`/`selected_space_ids` tolerate the missing Chainlit session context).
- **UI** (`src/static/js/assistant_settings.js` + modal in `index.html`): Settings button right of the space chips, shortcut `,` (Assistant tab only). Tabs: Model · Modes · Skills · Prompt · Composition. All changes are live (per-message reads) — no restart.
- **`instance/assistant/`** (via `chat/assistant_settings.py`, pure stdlib): the gitignored per-instance home — `system.md` (prompt override), `models.json`, `reasoning.json`, `skills/`. Created on access; the `instance/` gitignore rule covers it.

## Chat history (`chat/data_layer.py`)

- Chainlit's `SQLAlchemyDataLayer` is written **for Postgres** (array columns, JSON types); this module owns the **SQLite translation** and bootstraps the schema idempotently. Second SQLite DB at `instance/chainlit.db` (path from `settings.chainlit_db_path()`), next to the app DB.
- `SimplerSQLiteDataLayer(SQLAlchemyDataLayer)` subclasses upstream and fixes two Postgres⇄SQLite impedance gaps at the object/parameter seam (drift-proof — all actual SQL stays upstream):
  1. **`tags` array column**: upstream passes a raw Python `list` for `threads.tags`/`steps.tags` (Postgres array) which aiosqlite refuses to bind (`InterfaceError: unsupported type`). The subclass JSON-serializes lists on `update_thread`/`create_step` (re-applying `@queue_until_user_message` on `create_step` to preserve queueing + the test's `__wrapped__` access, delegating to upstream's unwrapped body to avoid double-queueing).
  2. **Round-trip**: every read path (`get_thread`, `list_threads`, `get_all_user_threads`, `get_step`, `get_favorite_steps`) calls `super()` then JSON-deserializes the `tags` TEXT back to a list via `_coerce_tags()` (safe on None/missing/already-list).
- `ensure_schema(db_path)` runs the additive `STEPS_ADDITIVE_COLUMNS` (`autoCollapse` from Chainlit 2.11) ALTER-TABLE back-fill on legacy DBs, since `CREATE TABLE IF NOT EXISTS` cannot evolve an existing table.
- **No storage provider**: element payloads (file uploads) are not persisted across thread reloads — fine for a single-user assistant, since uploaded file *content* is consumed into the conversation at message time via `chat/files.py`.

## Config (`.env`, read by `chat/settings.py`)

The assistant reuses the Flask app's `.env` exactly. Key vars: `AI_API_KEY` / `AI_API_BASE_URL` / `AI_MODEL` (provider), `CHAT_MODELS` (comma-separated model-picker fallback — `instance/assistant/models.json` wins when non-empty), `CHAT_MAX_TOKENS` (default 4096), `SECRET_KEY` (auth-secret derivation), `API_TOKEN` (sidecar bearer), `SIMPLER_MCP_URL` / `SANDBOX_MCP_URL` (pre-integrated tools), `CHAT_FILES_DIR` (shared workspace, default `/workspace` in compose), `CHAT_MAX_TOOL_ROUNDS` (default 8), `CHAT_WEB_TOOLS` (default `1`), `CHAT_MCP_SERVERS` (extra comma-separated MCP URLs), `CHAT_LOCAL_SANDBOX` (default `0`).
